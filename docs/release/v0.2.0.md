# bvm-lang v0.2.0 — The Execution Core

**The first release with a working VM.** v0.2.0 turns the scaffold into a running bytecode interpreter: a register instruction set (`Op`), an assembled-program type with automatic register sizing and branch back-patching (`Chunk`), a pooled-register interpreter (`Vm`), and a typed error set that keeps the loop panic-free on malformed input (`VmError`). The runtime value is `value-lang`'s eight-byte `Value`, wired here as the first-used dependency.

## What is bvm-lang?

A small, fast register-based bytecode virtual machine — the execution engine an interpreted language runs on once its source has been compiled to instructions. You assemble a `Chunk` of `Op` instructions over a constant pool and run it with a `Vm`, which returns a `Value`. Instructions are fixed-size decoded values dispatched through a single `match` loop; the crate is safe Rust (`unsafe` is forbidden) and treats bytecode as untrusted input.

## What's new in 0.2.0

### `Op` — a register instruction set

A register machine names the registers each instruction reads and writes, so `Add { dst, lhs, rhs }` does in one instruction what a stack machine needs four for. `Op` is a `Copy`, 8-byte, already-decoded value covering data movement (`Move`, `LoadConst`, `LoadNil`, `LoadBool`, `LoadInt`), arithmetic (`Add`, `Sub`, `Mul`, `Div`, `Rem`, `Neg`), comparison (`Eq`, `Ne`, `Lt`, `Le`, `Gt`, `Ge`), the logical `Not`, control flow (`Jump`, `JumpIfTrue`, `JumpIfFalse`), and termination (`Return`, `Halt`). It is `#[non_exhaustive]`, so later minor versions can add instructions without a breaking change.

### `Chunk` — assemble a program

```rust
use bvm_lang::{Chunk, Op, Value};

let mut chunk = Chunk::new();
let k = chunk.constant(Value::float(3.5)).expect("pool has room");
chunk.emit(Op::LoadConst { dst: 0, index: k });
chunk.emit(Op::LoadInt { dst: 1, val: 2 });
chunk.emit(Op::Mul { dst: 0, lhs: 0, rhs: 1 });
chunk.emit(Op::Return { src: 0 });
```

`emit` appends an instruction and returns its address; `constant` interns a `Value` and returns its pool index. The register file sizes itself — every emitted instruction widens it to cover the highest register it names, so a chunk built through this API can never address a register outside its own file. Forward branches (the exit of an `if`, the back-edge of a loop) are written with a placeholder target and fixed up later with `patch` once the landing address is known.

### `Vm` — run it

```rust
use bvm_lang::{Chunk, Op, Vm};

let mut chunk = Chunk::new();
chunk.emit(Op::LoadInt { dst: 0, val: 6 });
chunk.emit(Op::LoadInt { dst: 1, val: 7 });
chunk.emit(Op::Mul { dst: 0, lhs: 0, rhs: 1 });
chunk.emit(Op::Return { src: 0 });

let mut vm = Vm::new();
assert_eq!(vm.run(&chunk).unwrap().as_int(), Some(42));
```

`run` executes a chunk and returns its result. The register file is reset before each run and reused between runs, so a long-lived `Vm` executing many chunks reaches an allocation-free steady state. A `Vm` holds no reference to any chunk, so one chunk can be run on separate threads by separate VMs. `with_capacity` pre-sizes the file.

### `VmError` — faults, not panics

Bytecode is untrusted. Runtime faults (`TypeMismatch`, `DivideByZero`, `IntegerOverflow`) and structural faults in malformed bytecode (`BadRegister`, `BadConstant`, `BadJump`, `NoTerminator`) are all returned as typed errors — the dispatch loop checks every register, constant, and branch access, and the crate forbids `unsafe`, so no input drives it into a panic or undefined behaviour.

```rust
use bvm_lang::{Chunk, Op, Vm, VmError};

let mut chunk = Chunk::new();
chunk.emit(Op::LoadInt { dst: 0, val: 1 });
chunk.emit(Op::LoadInt { dst: 1, val: 0 });
chunk.emit(Op::Div { dst: 0, lhs: 0, rhs: 1 });
chunk.emit(Op::Return { src: 0 });
assert_eq!(Vm::new().run(&chunk), Err(VmError::DivideByZero));
```

### Numeric semantics

Arithmetic and ordering form one numeric tower: integer-with-integer stays integer and is overflow-checked (a fault, never a silent wrap), while any float operand promotes the result to float. Integer division/remainder by zero errors; float division by zero follows IEEE-754. Comparisons and `Not` produce booleans, and every branch condition must be a boolean — there is no implicit truthiness.

### `serde` support

Behind the `serde` feature, `Op` and `Chunk` derive `Serialize`/`Deserialize`, so compiled bytecode can be written to any serde format and reloaded to an identical, runnable program.

### Dependencies

`value-lang = "1"` is wired as the operand type — the "first used" dependency for this phase. `gc-lang` and `ir-lang` are deliberately not yet wired: the core VM executes register bytecode over `Value` and has no heap-object or IR path to use them from. They land in additive 1.x work (GC when heap-allocated objects arrive, IR when direct execution or `ir -> Chunk` lowering arrives). The deferral is recorded in [`dev/ROADMAP.md`](../../dev/ROADMAP.md) per the anti-deferral rule.

## Performance

Local Criterion means (`cargo bench`, Windows x86_64, Rust stable), reusing one `Vm`:

- Straight-line expression (10 instructions): **~14 ns/run** — ~1.4 ns per instruction.
- Counted loop `sum 1..=1000` (~5 000 instructions executed): **~7 µs/run** — ~1.4 ns per instruction dispatched.

Numbers are hardware-dependent; `cargo bench` reproduces them on your target.

## Breaking changes

**None** relative to v0.1.0 (the scaffold had no public API). This release introduces the initial public surface. It remains subject to change across the 0.x series and is frozen at 1.0.0.

## What's next

- **1.0.0 — API freeze.** Stabilize and freeze the public surface, mark `docs/API.md` stable with the SemVer promise, and confirm the full test + benchmark suite green on all three platforms.
- **Additive 1.x** — function calls / multiple chunks, an execution fuel limit for hard bounds on untrusted programs, and `gc-lang` / `ir-lang` wiring. Each is non-breaking and independent of the frozen core.
